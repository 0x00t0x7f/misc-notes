## spec技术规范
在每个软件实现元素（组件/服务）的代码仓库中，建立统一的 .codespec/默认目录承载所有规格产物。

+ SPEC.md：.codespec/SPEC.md 组件的全量功能规格文件（what & why），回答 系统应该表现出什么业务行为，是业务规则、数据约束、DFX红线的权威定义。
+ DESIGN.md：.codespec/DESIGN.md 组件的全量设计文件（how），回答 用什么技术手段实现spec中定义的业务行为，是架构、数据模型、接口、算法、缓存策略的权威定义。
+ SE：系统工程师，负责需求澄清与业务规格。
+ MDE：模块设计师，负责软件实现元素级设计

**物理形式与目录结构** 
```
repo-root
└── .codespec                        # codespec默认根目录（每个项目唯一）
    ├── SPEC.md                      # 全量功能规格（what & why，必需）
    ├── DESIGN.md                    # 全量实现设计（how，必需）
    ├── changes                      # 增量变更工作区（必需）
    │   └── AR-ID                    # 每个AR对应一个变更目录
    │       ├── proposal.md          # 阶段1：需求澄清
    │       ├── delta-spec.md        # 阶段2：spec增量设计
    │       ├── delta-design.md      # 阶段3：design增量设计
    │       └── tasks.md             # 阶段4：任务拆解
    └── archive                      # 归档区（必需）
        └── YYYYMMDD-AR-ID           # 完成后归档的变更副本
```

**SPEC.md编写规范-数据约束编码规范** 
+ 标识符： 订单ID必须全局唯一； 订单ID使用雪花算法生成，存储为BIGINT
+ 数量：商品数量必须为正整数；商品列表使用JSON字段存储于orders表
+ 精度：金额精度为小数点后两位； 金额使用 DECIMAL(18,2)类型
+ 状态：枚举值范围 CREATED/PAID/CLOSED；状态字段建立索引以优化查询


**DESIGN.md编写规范** 

DESIGN.md必须遵循以下一级章节顺序

|标题|关注点|
|---|---|
|设计概述|设计目标、设计约束|
|系统架构|架构概览、模块职责、技术栈|
|数据模型|ER图、表结构、索引设计|
|接口设计|接口清单、请求响应详设、错误码|
|核心流程设计|关键算法的伪代码与复杂度分析|
|缓存设计|缓存策略、一致性保障|
|异常处理设计|异常分类、熔断降级|
|监控与日志|监控指标、日志规范|
|安全设计|认证、授权、数据保护方案|

+ DESIGN.md必须以SPEC.md中定义的业务行为为输入，禁止描述任何Spec中未约束的业务能力。
+ 当DESIGN.md的某项技术决策与SPEC.md中的业务规则冲突时，必须以Spec为准修正Design，禁止反向修改Spec以迁就Design
+ 代码服从Spec，Design也服务Spec，这是规格权威性的体现。
+ DESIGN.md中的每项设计决策应当能够追溯到SPEC.md中的至少一条业务规则或DFX约束。

**Spec&Design的严格界定** 

SPEC.md管辖范围：
1. 业务流程与状态流转逻辑
2. 权限约束与数据有效性校验规则
3. 与外部角色、外部系统的交互行为契约
4. DFX约束红线（性能、可靠性、安全、可维护性、兼容性）
5. 领域对象的逻辑约束（唯一性、枚举、范围、精精度）

严禁写入SPEC.md的内容（属于DESIGN.md）
1. 数据库表结构、字段类型、索引设计
2. 类与接口的内部设计、命名、继承关系
3. 函数调用链与算法实现
4. 缓存策略、消息队列主体、序列化格式等技术选型
5. 第三方库的版本号、配置参数

+ 业务约束必须按”是否随技术栈或实现方案变更而失效”进行归属判定：不随技术栈变更而失效的约束写入SPEC.md，随技术栈变更而失效的约束写入DESIGN.md（例如”订单ID必须全局唯一”不随技术栈变化而失效，属于Spec；“订单ID使用雪花算法生成”会随ID生成方案变更而失效，属于Design）。
+ Spec评审中发现的实现细节内容必须从SPEC.md中移除，并迁移到DESIGN.md对应章节。
+ DESIGN.md中描述的能力必须能够追溯到SPEC.md中已定义的业务规则，禁止出现SPEC.md未约束但DESIGN.md已实现的能力。
+ 避免实现侧自行扩展未经规格定义的功能，导致规格与实现不一致。

**图表与格式规范** 
+ 所有Spec/Design文档必须使用标准markdown语法编写
+ 所有图表必须使用PlantUML代码块绘制，严禁使用二进制图片文件、外部图床链接或截图嵌入
+ PlantUML是文本格式，可被Git diff追踪，可被AI解析与生成。二进制图片破坏了”规格即代码”的核心原则
+ 交互流程图必须使用participant与actor明确标注角色，禁止使用未命名的匿名节点
+ PlantUML图表统一使用!theme plain主题，确保渲染风格一致
+ 单个Markdown文件的有效内容应当控制在2000行以内；超出时，应当审查是否混入了不属于该文件的内容

## Spec体系与三层设计

**三层设计体系** 
软件设计是一套自顶向下，逐层细化的体系。

|层次|名称|责任人|codespec对应产物|
|---|---|---|---|
|第一层|产品级软件实现架构设计|软件架构师|产品级 SPEC.md+DESIGN.md（跨组件公共框架）|
|第二层|软件实现元素级设计|MDE|组件级SPEC.md+DESIGN.md（长期维护的基线）|
|第三层|AR级实现设计|开发人员|.codespec/changes/AR-ID/下的delta流程|

+ 每一层设计必须以上一层的输出作为输入基线，禁止跨层设计或绕过上层约束
+ 当AR级设计过程中发现新的抽象机会或架构改进点时，必须反馈刷新元素级或产品级的全量文档，保持设计体系的一致性
