# 引言
 有时候性能优化花了很大的时间精力，效果却没有达到预期。
 在性能优化中，架构优化通常获得的收益最大但是付出的成本也最高，指令级的优化付出的成本低，但是收益不是很显著。
 算法优化常常可以在成本和收益之间取得一个平衡，在不用花费大量重构成本的情况下可以得到可观的性能提升。如果存在算法优化的空间，性能优化应当首先考虑算法优化，在充分进行串行程序算法优化后，再进行并行计算优化而充分利用多核的能力。

 # 常用优化算法

 算法优化是在不改变现有接口的情况下，使用更高效的算法替换现有算法。同时包含着数据结构设计优化。

 + 冷热编排：定义数据结构的时候，将经常范文的热数据尽量排列在一起，有利于cache预取的时候提高cache命中率。
 + 成员变量读写分离
 + 延迟计算：最近用不上的变量先不要初始化。变量初始化是一个好的编程习惯，但在性能优化的时候，有可能就适得其反。
 + 提前计算：有些变量或者函数在多个地方被调用，且会多次使用，但是输出的结果是一致的。这种情况下最好提前计算出结果，以后再引用，避免每次调用时重新计算。
 + 字节对齐：如果不按照平台要求对数据存放进行对齐，会在存取效率上带来损失； 申明结构体时，尽量大数据类型在前，小数据类型在后，这样会节省一些空间，另一方面可以更好的满足处理器对齐要求。
 + 优化乘除法和模余：当数据量不大的时候，一些线性数据结构比如数组、链表并不会过多影响性能。当数据量达到成千上万时候，线性算法常常会推高CPU占用率，二分查找比线性查找高效很多。虽然二分查找需要将数据先进行排序，但是带来的收益足以抵消初始排序的代价。
 + 查询优化：最常用的是增加索引，用额外的存储空间换取查询时间，提高查询效率。经常使用的索引方法包括：
   - 哈希表与一致性哈希：容量充足的哈希表可以将查找时间复杂度从O(N)降到O(1)。 哈希表的一个问题是哈希表容量不容易确定。不同的客户数据量不同，不用的机器类型内存配置也不同，因此很难确定一个高低兼顾的哈希表大小。容量太大，容易造成内存占用过多；容量太小，容易造成哈希冲突。在分布式系统中，当使用哈希算法做负载均衡的时候，如果节点增加或者减少，会影响所有当前处理对象的哈希值，从而造成处理节点的不一致。 一致哈希算法通过将处理对象和处理机器分别散列到一个虚拟区间环来最大限度的减少增减节点所造成的影响。
   - 二叉搜索树与红黑树：二叉搜素树在空间复杂度和时间复杂度之间取得了一个较好的平衡；通常二叉搜素树不需要像哈希表那样在初始阶段就占用很多内存。二叉搜素树有很多种，包括平衡二叉搜素树、红黑树等。中红黑树在C++、Java等编程语言底层数据结构中有广泛的应用。红黑树是一种不完全平衡的二叉搜索树。
红黑树的关键特性是从根到叶子的最长的可能路径不多于最短的可能路径的两倍长。结果是这个树大致上是平衡的。因为操作比如插入、删除和查找某个值的最坏情况时间都要求与树的高度成比例，这个在高度上的理论上限允许红黑树在最坏情况下都是高效的，而不同于普通的二叉查找树，所以它的操作有着良好的最坏情况运行时间。
   - 跳表：跳表是一种多层结构的有序链表，插入一个值时有一定概率“晋升”到上层形成间接的索引。跳表更适合大量并发写的场景，可以认为是随机平衡的二叉搜索树，不存在红黑树的再平衡问题。Redis强大的ZSet底层数据结构就是哈希加跳表。
   - 堆：堆是用数组实现的二叉树；堆分为两种：最大堆和最小堆，两者的差别在于节点的排序方式。在最大堆中，父节点的值比每一个子节点的值都要大。在最小堆中，父节点的值比每一个子节点的值都要小。
   - 其他经典算法和多核优化算法：还有很多高效的数据结构和算法，包括平衡多路搜索树（B-Tree）、叶节点相连的平衡多路搜索树（B+ Tree）、前缀树（Trie Tree）等。其中在路由匹配场景下应用广泛的基数树（Radix Tree）就是前缀树的一种。经典排序算法中的快速排序虽然是一种不稳定排序方法，但是它的综合效率很高、应用也十分广泛。针对目前普遍使用的多核处理器，国内外的学者也做了算法优化的研究。比如FAST(fast architecture sensitive tree)， CST-tree(cache sensitive T-tree)，归并排序, 双调排序, 基数排序等。这些算法通过优化程序局部性、减少cache 访问冲突、提高线程并行度、充分利用单指令多数据流(single instruction multiple data, SIMD)并行和带宽优化等几方面提高算法在多核CPU上运算效率。

# 并行算法设计方法
当在串行模式下，如果算法优化已经到了没有潜力可挖的境地，就应当考虑采用多实例（多进程）或者多线程的设计方法，充分挖掘CPU多核能力。

通常并行算法的设计分为几个部分：划分、通信、结果合并和负载均衡。

多线程模式下线程切换负担比较小，是很多并行优化的选择，但是它对编程人员的要求也较高，需要仔细设计数据的共享与冲突。

多线程重构重点是如何把原来单线程处理的任务和数据进行划分。

任务划分的方法主要有两种：一种是水平划分；另一种是垂直划分。

水平划分是指拆分的线程实现同样的逻辑功能，但是处理不同的数据。这类似于数据库的水平分片。水平划分最重要的是减少线程间的共享数据，从而减少线程互锁的概率。水平划分的最大好处是具有良好的扩展性。系统可以通过增加并发线程数量来改善系统性能。
如果不能做到全并行，也可以使用垂直划分，设计流水线式的并发结构。比如一个数据需要经过主处理逻辑、硬件下表和数据库存储。每一级流水线可以设计一个线程。然后可以把资源占用最高的线程比如主处理逻辑继续划分成并行线程。这样比较适合有些逻辑无法分开的场景，但是这种情况下需要设计比较复杂的状态机，协调执行结果。

工作任务的分发是拆分线程的关键。除了可以用Transaction ID这种实体ID来分发之外，还可以用组合的逻辑ID来分发或者使用分发表来分发，这样可以达到更灵活的负载均衡效果。

在高性能场景下，任务之间的通信可以多采用队列进行通信。队列通信可以起到平滑负载、削峰填谷的作用。为了减小锁的开销，可以使用无锁队列。为了使高优先级的消息得到及时处理，可以使用优先级队列进行处理。

多线程并行还要注意函数可重入性和线程安全。线程安全是指当多个线程并发/并行调用该函数时，它依旧可以产生正确的结果。

通常线程安全的函数有以下几个特征：
+ 为共享变量的读写提供保护。一般使用了C中static变量的函数就不是线程安全的。
+ 没有多个函数共享的状态。C标准库中的rand函数不是线程安全的。因为它需要在多次调用见保持中间状态，一旦并行调用时，某些线程就可能读到了另一个线程的中间运算数据。
+ 没有返回指向静态变量的指针。从并行线程调用这类函数时，一个线程使用的结果很可能被另一个线程覆盖。
